Key Sizes

  • AES: 128, 192, and 256 bit keys are supported for all uses of AES, except for PBKDF2-AES-CMAC-PRF128 which uses 128 bit keys.
  • RSA: 1, 1.5, 2, 3, 4, 6, and 8 kbit keys are supported for all uses of RSA.
  • NIST curves: P-224 (aka secp224r1), P-256 (aka secp256r1), P-384 (aka secp384r1) and P-521 (aka secp521r1) support 224/256/384/521 bit keys.
  • Twisted Edwards curves: Ed25519 and Ed448 support 255/448 bit keys.
  • Montgomery curves: X25519 and X448 support 255/448 bit keys.
  • SRP-6: 3072 bit keys are supported.

The availability of a function depends on your license agreement.

Function Types

ocrypto supports these three types of functions:

Single-part

Function is executed in a single call, using only the stack.

Stack-optimized

Function is executed in a single call, using minimal stack space. Additional memory is provided by caller in a context object. The function names end in ctx.

Incremental

Function is executed in multiple calls, to allow for streams of data to be processed in a piecemeal fashion. Additional memory is provided by caller in a context object. The functions are split into sets of functions whose names end in init, final and update.

RAM Requirements

ocrypto does not use static memory.

ocrypto does not use dynamic memory allocation - no heap is required.

The following table indicates how much memory is used on the stack:

Function and type Required size (bytes)
Single-part RSA < 7,900
Single-part SRP < 3,900
Single-part Ed25519 < 1,300
Single-part (other) < 1,000
Stack-optimized SRP < 400
Stack-optimized (other) < 300
Incremental (any) < 300

For stack-optimized and incremental functions, the caller must provide pre-allocated context object, of similar size as stack space for single-part functions. The application may allocate these context objects statically or in a heap.

Build Options

ocrypto supports the following build option:

  • OCRYPTO_NO_CACHE: Define this option only if you are sure that your system (microcontroller or external memory) does not use a data cache and side-channel attacks based on radiation measurements are considered acceptable. Using this option with a data cache makes the system prone to side-channel attacks. For more information, see the Side-Channel Safety Statement. By default, this option is not defined.

Note: Only data caches are critical; instruction caches are not an issue regarding side-channel resistance.
